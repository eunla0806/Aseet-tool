## 0. 이미지 타입 한눈에 보기 (용어 요약 테이블)

> 이 표는 파이프라인에서 사용되는 8가지 핵심 이미지 타입의 관계를 한눈에 정리한 참조용 요약입니다.

| 용어 | 소스 / 생성 방법 | 주요 역할 | 관련 단계 |
| :--- | :--- | :--- | :---: |
| `external_ai_stand1_draft` | 외부 AI 이미지 생성 도구로 직접 생성 | STAND1 4프레임 의상 디자인 초안 (2x2 그리드, 860×1360 권장, 단색 배경) | 5-A |
| `clothing_only_anchor` | draft에서 배경·스킨톤 색상 제거 (Python 스크립트) | 투명 배경 앵커 의상 레이어 (다운스케일하여 `43 × 68` 획득) | 5-B |
| `body_anchor_preview` | `clothing_only_anchor` + `body_reference` 합성 | 초안 의상의 사전 피팅 검수 이미지 (QA 통과 필수, 120×120) | 5-B |
| `temp_base` | `body_reference` + `clothing_only_anchor` 합성 | ComfyUI API 호출 시 인페인팅 입력 이미지로 주입하는 임시 베이스 캐릭터 이미지 | 5-C~ |
| `clothing_only_source` | 기존 샘플 의상 레이어 파츠 직접 합성 (AI 처리 없음) | 무결점 정답 레이어 (LoRA 학습·마스크 생성·QA 비교용) | 4.5 |
| `body_reference` | 바디 템플릿 원본 시트 | 순수 기준 바디 템플릿 (맨몸 베이스, 모든 프리뷰/합성의 배경재) | 전 단계 |
| `raw_comfy_output` | ComfyUI API 출력 (KSampler 직후) | 날것의 중간 결과 (바디+의상 혼합 상태) | 5-C~ |
| `clothing_only_candidate` | `raw_comfy_output ∩ clothing_mask` (픽셀 교집합) | 신규 생성 의상 후보 (투명 배경 · 최종 품질 평가 대상) | 5-C~ |
| `body_preview` | `clothing_only_candidate` + `body_reference` 합성 | 최종 착용 피팅 검수 프리뷰 이미지 | 5-C~ |

> **용어 구분 핵심 포인트**  
> - `clothing_only_source` = 기존 원본 샘플에서 추출한 **정답 레이어** (불변)  
> - `clothing_only_anchor` = 이번에 새로 만들 신규 의상의 **디자인 앵커** (외부 AI 초안 기반)  
> - `body_reference` = 의상이 전혀 없는 **순수 맨몸 바디 템플릿** (배경용)  
> - `temp_base` = ComfyUI 인페인팅 생성을 위해 `body_reference` 위에 옷을 얹은 **입력용 합성 이미지**

## 1. 용어 정리

### template_grid

MapleStory Worlds 아바타 바디 기준 템플릿.  
모든 의상 생성의 기준이 되는 원본 그리드이다.

---

### clothing_grid

template_grid와 같은 좌표 구조를 가진 의상 전용 시트.  
최종적으로는 바디 없이 의상만 남아야 한다.

---

### Inpainting Mask

AI가 수정할 영역을 제한하는 마스크.  
의상 영역만 칠하고 얼굴, 손, 발, 피부는 보호한다.

본 프로젝트에서 가장 중요한 기본 요소이다.

---

### Canny ControlNet

template_grid의 외곽선과 포즈를 유지하는 보조 모델.  
바디 실루엣 고정에 사용한다.

---

### LoRA

샘플 의상 데이터의 도트 스타일을 학습하는 모델.  
좌표 정렬이 아니라 스타일 고정에 사용한다.

---

### IP-Adapter

특정 참고 의상의 색감, 분위기, 디자인 방향을 반영하는 보조 모델이다.

---

### 후처리

생성 결과에서 바디 픽셀을 제거하고, 의상만 남기는 과정이다.

---

### external_ai_stand1_draft

외부 AI 이미지 생성 도구를 활용해 제작한 최초의 STAND1 4프레임 의상 디자인 초안. 신규 의상의 색감, 장식, 실루엣을 시각화하고 최종 확정하기 위한 핵심 디자인 앵커용 초안입니다.

---

### clothing_only_anchor

`external_ai_stand1_draft` 이미지에서 바디(스킨)와 불필요한 배경 영역을 정교하게 지워낸 투명 배경 앵커 의상 레이어. ComfyUI IP-Adapter의 reference 및 생성 가이드이자 위치 정렬의 절대적인 기준점으로 사용됩니다.

---

### body_anchor_preview

`clothing_only_anchor`를 `body_reference` 위에 오버레이하여 생성한 사전 피팅 검수 이미지. 초안 의상이 실제 바디 규격 및 포즈에 정확히 피팅되는지 사전 확인하는 수단입니다.

---

### clothing_only_source

기존 샘플 의상(예: `05_멜로디 소녀` 구조 - stand1_0, walk1_0 등 프레임 단위 폴더로 나뉘어 있고 각 폴더 내에 바디 및 옷 레이어가 분리되어 저장된 구조)에서 **순수 의상 파츠 레이어만 직접 1:1 합성**하여 완성한 완벽한 정답 의상-only 레이어 (학습, 레퍼런스, 마스크 생성 및 AI 결과 비교 대조군 역할). AI 생성이나 바디 차감 등의 처리를 타지 않은 무결점 원본 에셋입니다.

---

### body_reference

의상이 전혀 포함되지 않은 **순수 기준 바디 템플릿(맨몸 베이스)** 시트. 모든 피팅 프리뷰 이미지(`body_anchor_preview`, `body_preview`)의 배경재로 사용되며, `temp_base`를 만들기 위한 순수 바디 소스 역할을 합니다. (AI 생성 대상이 아님)

---

### temp_base

`body_reference` (순수 기준 바디 템플릿) 위에 `clothing_only_anchor` (또는 타겟 의상 레이어)를 합성하여 생성한 **ComfyUI 생성 입력용 임시 베이스 캐릭터 이미지**. ComfyUI API 호출 시 인페인팅 소스(`image`)로 주입되어 AI가 의상 영역을 재창작 또는 동작 확장할 수 있도록 기초적인 형상을 제공하는 가이드 레이어 역할을 합니다.

---

### raw_comfy_output

ComfyUI API 호출 후 획득한 날것의 인페인팅 완료 원본 이미지 (바디와 생성 의상이 합쳐져 있는 중간 단계 에셋).

---

### clothing_only_candidate

`raw_comfy_output`에서 `clothing_mask` 영역만을 투명 배경으로 오려내어 완성한 순수 의상-only 레이어 후보군. 최종 품질 평가의 핵심 지표.

---

### body_preview

`clothing_only_candidate` 의상 레이어를 `body_reference` 위에 다시 오버레이하여 생성 및 결합한 프리뷰 이미지. 실제 캐릭터가 착용한 피팅(Fitting) 핏과 어긋남을 QA 검수하는 용도.

---

## 2. 기본 전략 우선순위

### 1순위: 후처리 및 마스크 추출 (의상-only 레이어 분리)

최종 목표인 바디 제외 투명 의상 레이어를 획득하기 위해, `raw_comfy_output ∩ clothing_mask` 공식을 통한 픽셀 오려내기를 수행하고 `body_preview`로 착용 정렬 상태를 QA하는 근본적인 후처리 필터입니다.

---

### 2순위: Inpainting Mask

옷이 들어갈 영역을 제한하고 얼굴, 피부, 손, 발을 보호하는 1차 차단막이자 결과물 추출 마스크입니다.

---

### 3순위: Canny ControlNet

template_grid의 포즈와 실루엣을 단단히 고정하여 AI가 의상 형태를 무너뜨리지 않도록 지탱합니다.

---

### 4순위: LoRA

도트 의상 스타일(외곽 검은 테두리선 두께, 명암 디테일)을 일관되게 복원하는 스타일 고정 장치입니다.
> [!IMPORTANT]
> **LoRA의 기술적 한계 명시**
> LoRA는 오로지 화풍 및 스타일의 개선 장치일 뿐이며, 바디 침범, 얼굴 뭉개짐, 픽셀 겹침(Overlap) 또는 바디 복구 등 **공간적/구조적 물리 문제를 해결하는 수단이 절대 아닙니다.** 
> 바디 보존 및 오염 방지는 오직 `clothing_mask`와 `preserve_mask` 간의 수학적 차집합 보정(`preserve_mask_final = raw_preserve_mask - clothing_mask`) 및 `clothing_mask` 영역을 통한 교집합 추출 후처리로만 해결합니다.

---

### 5순위: IP-Adapter

참고 의상의 고유 색감 및 장식 분위기를 전이하는 보조 모델입니다.

---

## 2.1 외부 AI와 ComfyUI의 역할 분담 기술 메모

### 2.1.1 왜 역할 분담이 필수적인가? (ComfyUI 단독 생성의 한계)
* **ComfyUI 단독 창작의 한계**: KSampler만을 구동하여 맨몸 바디 템플릿과 텍스트 프롬프트만으로 새로운 의상 디자인을 창작하려 할 경우, 다리나 하체 영역에 의도치 않은 살구색 노이즈가 침범(오염)하거나, 치마/바지 고유의 실루엣이 제대로 묘사되지 않고 번지는 결함이 빈번히 발생합니다.
* **하이브리드 전략의 이점**: 
  - 공홈 이미지 생성 도구 및 기타 이미지 생성 서비스 등 **외부 AI 이미지 생성 도구**를 사용하여 최상의 디자인 컨셉과 명확한 앵커 실루엣을 가진 **STAND1 4프레임 초안(`external_ai_stand1_draft`)**을 선제 조율하여 획득합니다.
  - 이 초안을 앵커(`clothing_only_anchor`)로 확정한 뒤 ComfyUI의 레퍼런스로 주입하면, 처음부터 의상을 무작위로 그릴 필요가 없어지므로 **디자인 품질과 타 동작(walk, jump, sit 등) 확장 시의 비주얼 일관성을 비약적으로 높일 수 있습니다.**

### 2.1.2 외부 AI 초안 적용 시의 절대 원칙
외부 AI 도구로 완성한 초안 이미지 역시 최종 스프라이트 시트 에셋으로 즉시 사용할 수 없습니다. 반드시 아래 로컬 정형화 단계를 완벽히 거쳐야 합니다.
1. **정밀 오려내기 (`clothing_only_anchor` 빌드)**: 초안 의상 영역 내에 묻어 있는 바디(스킨) 픽셀과 배경을 100% 제거하고 순수 의상만 남은 투명 배경 레이어로 분리해야 합니다.
2. **사전 피팅 검수 (`body_anchor_preview` 조립)**: 분리된 앵커 의상을 실제 바디 템플릿(`body_reference`) 위에 정렬 배치하여, 어깨선, 치마의 세로 길이, 다리 노출 면적 및 착용 피팅 위치가 도트 규격에 물리적으로 딱 맞아떨어지는지 수동/자동 검수를 통과해야만 ComfyUI 타 동작 확장의 레퍼런스로 활용 가능합니다.

## 2.2 `build_anchor_from_draft.py` 기술 명세

### 2.2.1 개요

`external_ai_stand1_draft` (2x2 그리드 PNG)에서 배경과 바디 스킨 픽셀을 자동 제거하여 투명 배경 앵커 의상 레이어(`clothing_only_anchor`)를 추출하고, 다운스케일 및 `body_anchor_preview`까지 일괄 생성하는 로컬 파이프라인 스크립트입니다.

> [!IMPORTANT]
> **구현 상태: 미구현 (STEP 5-B 진입 전 반드시 구현 필요)**  
> 이 스크립트가 완성되기 전까지는 STEP 5-B를 진행할 수 없습니다.

---

### 2.2.2 입력 조건

| 항목 | 표준 권장 사양 (Standard) | 대체 근사 사양 (Fallback) |
| :--- | :--- | :--- |
| **전체 해상도** | **`860 × 1360` px** | **`816 × 1290` px** |
| **개별 셀 해상도** | **`430 × 680` px** (10배 정밀 스케일) | **`408 × 645` px** (근사 비율 스케일) |
| **파일명** | `stand1_draft.png` (4개 프레임이 포함된 단일 2x2 그리드 시트) | Same |
| **배경** | **단색 단일 배경 필수** (흰색 또는 밝은 단색 권장, 그리드 간 경계선/그라데이션 금지) | Same |
| **형식** | PNG 또는 JPEG | Same |

> **💡 외부 AI 이미지 생성 시 해상도 처리 원칙**
> - **권장 사양 (`860 × 1360` px)**: 최종 도트 규격인 `43 × 68` px의 정확히 10배 정수 스케일이므로 다운스케일 시 화질 유실이 가장 적습니다.
> - **대체 사양 (`816 × 1290` px)**: 일부 생성 AI 플랫폼(예: Midjourney 등)의 종횡비 제약으로 인해 표준 해상도 출력이 불가능할 경우 사용하는 근사 해상도입니다. 이 경우에도 `build_anchor_from_draft.py`가 최종 단계에서 Nearest Neighbor 보간을 사용해 최종 타겟인 `43 × 68` px로 강제 재보정 정밀 정렬을 수행하므로 에셋 규격 무결성이 유지됩니다.
> - **레이아웃**: **2x2 그리드 레이아웃** (상단 좌: stand1_0, 상단 우: stand1_1, 하단 좌: stand1_2, 하단 우: stand1_3)
> - **배경**: 단순 흰색 또는 단색 단일 배경 (그림자/경계선/텍스트 없음)  
> - **구도**: 각 프레임의 크기와 센터 정렬이 균일해야 함  

---

### 2.2.3 처리 알고리즘

```txt
[Step 1] 2x2 그리드 분할 (Grid Splitting)
  - 입력받은 `stand1_draft.png` 해상도를 자동 감지하여 아래 분할 좌표 기준으로 각각 프레임 이미지 분할:
    
    * CASE A: 860 × 1360 (표준 권장) -> 430 × 680 크기 4장으로 분할
      - stand1_0_draft: x=[0, 430], y=[0, 680]
      - stand1_1_draft: x=[430, 860], y=[0, 680]
      - stand1_2_draft: x=[0, 430], y=[680, 1360]
      - stand1_3_draft: x=[430, 860], y=[680, 1360]
      
    * CASE B: 816 × 1290 (대체 근사) -> 408 × 645 크기 4장으로 분할
      - stand1_0_draft: x=[0, 408], y=[0, 645]
      - stand1_1_draft: x=[408, 816], y=[0, 645]
      - stand1_2_draft: x=[0, 408], y=[645, 1290]
      - stand1_3_draft: x=[408, 816], y=[645, 1290]

[Step 2] 배경 제거 (Flood Fill 방식) - 각 프레임 개별 적용
  - 각 프레임 이미지의 네 모서리 색상을 샘플링하여 배경 기준색 확정
  - 기준색과 허용 오차(tolerance=30) 이내인 픽셀을 Alpha=0으로 투명화
  - 결과: 캐릭터 실루엣만 남고 배경 제거됨

[Step 3] 스킨톤(피부색) 픽셀 제거
  - MASK_SPEC.md 2.1절의 스킨톤 감지 기준 적용:
      R >= 220 AND 150 <= G <= 255 AND 80 <= B <= 240 AND R >= G > B
  - 해당 범위에 속하는 픽셀은 Alpha=0으로 투명화
  - 결과: 바디 피부, 얼굴, 손발 픽셀 제거됨

[Step 4] 고립 픽셀 정리 (선택 옵션)
  - Alpha가 남아있는 픽셀 중 주변 3×3 내 불투명 이웃이 0인 고립 픽셀 제거

[Step 5] clothing_only_anchor 저장 (분할된 원본 크기 투명 PNG)
  - 저장 경로: outputs/anchor/{clothing_name}/stand1_{n}_anchor_original.png

[Step 6] 다운스케일 → clothing_only_anchor_scaled 저장 (exactly 43×68)
  - 보간: Nearest Neighbor (PIL.Image.NEAREST)
  - 저장 경로: outputs/anchor/{clothing_name}/stand1_{n}_anchor.png

[Step 7] body_anchor_preview 생성 (120×120)
  - 기존 MASK_SPEC.md 3.2절의 오프셋 공식으로 120px 셀 위에 배치
  - 배경에 body_reference (순수 기준 바디 템플릿)를 깔고, 그 위에 anchor_scaled를 오버레이
  - 저장 경로: outputs/anchor/{clothing_name}/stand1_{n}_anchor_preview.png
```

---

### 2.2.4 출력 파일

| 파일명 | 해상도 | 용도 |
| :--- | :--- | :--- |
| `stand1_{n}_anchor_original.png` | 430×680 또는 408×645 | 고해상도 앵커 원본 (보존용) |
| `stand1_{n}_anchor.png` | 43×68 (정밀 고정) | 실제 사용 앵커 (ComfyUI 레퍼런스 및 다운스케일 완료 의상 레이어) |
| `stand1_{n}_anchor_preview.png` | 120×120 | body_anchor_preview (순수 바디 템플릿과의 피팅 QA 검수용 프리뷰) |

### 2.2.5 주의사항

- 외부 AI 생성 시 배경이 단색이 아니거나 그라데이션이 있을 경우 Step 1의 flood fill이 정확히 작동하지 않습니다. **단색 배경 생성 요청은 필수 조건**입니다.
- 의상 색이 스킨톤 범위와 겹치는 경우(살구색 계열 의상 등) Step 2에서 의상 픽셀이 함께 제거될 수 있습니다. 이 경우 스킨톤 허용 범위를 좁혀 tolerance를 수동 조정해야 합니다.
- Nearest Neighbor 다운스케일은 도트 픽셀 아트 고유의 날카로운 테두리를 보존하기 위해 필수입니다. Bilinear, Lanczos 등 다른 보간은 사용하지 않습니다.

## 12. 해상도 전략 (Resolution Strategy)

이 항목은 프로젝트의 가장 큰 기술 리스크 중 하나이다.

### 문제

Stable Diffusion 계열 모델은 512×512 이상의 이미지에 최적화되어 있다.  
MapleStory Worlds 스프라이트의 프레임 한 칸은 약 30~60px 수준이다.  
이 크기에서 직접 인페인팅하면 세일러 라인, 리본, 프릴 같은 디테일이 뭉개진다.

### 해결 방향

개별 프레임을 인페인팅하기 전에 업스케일하고,  
생성 후 다시 원래 크기로 다운스케일한다.

```txt
원본 프레임 (약 40×60px)
→ 4~8배 업스케일 (약 160×240 ~ 320×480px)
→ Inpainting 수행
→ 원래 크기로 다운스케일
→ pixel-perfect 보정
```

### 주의사항

- 업스케일 시 Nearest Neighbor 보간을 사용해야 도트 픽셀이 깨지지 않는다.
- 다운스케일 후 도트가 안티앨리어싱되지 않도록 주의한다.
- 업스케일 배율은 테스트를 통해 정해야 한다. 4배부터 시작한다.
- 이 흐름은 STEP 5 테스트 시 반드시 검증해야 한다.

## 13. 처리 단위: 그리드 vs 개별 프레임

### 문제

template_grid는 124프레임이 한 장의 PNG에 빽빽하게 배치되어 있다.  
이 큰 시트를 통째로 인페인팅에 넣으면 AI가 프레임 경계를 인식하지 못하고,  
의상이 옆 프레임으로 번지거나 프레임 간 영향을 받는다.

### 해결 방향

그리드를 개별 프레임으로 분리 → 프레임별로 처리 → 다시 그리드로 조립한다.

```txt
template_grid.png
→ analyze_template.py로 프레임 좌표 추출
→ 개별 프레임 PNG 분리 (frames/ 폴더)
→ 프레임별 업스케일
→ 프레임별 Inpainting
→ 다운스케일
→ 결과를 다시 그리드로 조립
→ clothing_grid.png 완성
```

### 필요한 스크립트

- `analyze_template.py`: 그리드 규격 분석, 프레임 좌표 추출
- `split_grid.py`: 그리드를 개별 프레임으로 분리
- `assemble_grid.py`: 처리된 프레임을 다시 그리드로 조립

### 폴더 구조 보완

```txt
E:\Aseet tool\
├── frames\              # 개별 프레임 PNG 저장
│   ├── template\        # template_grid에서 분리한 바디 프레임
│   ├── masks\           # 프레임별 인페인팅 마스크
│   ├── generated\       # AI 생성 결과 (바디 + 의상)
│   └── clothing\        # 바디 제거 후 의상만 남은 프레임
```

## 15. 바디 제거 색상 허용 오차 (Body Subtraction Tolerance)

### 문제

`remove_body.py`의 "원본 바디와 동일한 픽셀을 빼기" 전략은  
AI가 바디를 원본과 100% 동일하게 남겨줬을 때만 완벽하게 작동한다.

하지만 인페인팅은 마스크 경계 근처의 피부색을 미묘하게 변형시킬 수 있다.  
denoise가 낮아도 바디 색상이 RGB 기준 1~5 정도 달라질 수 있다.

이 경우 픽셀이 "동일하지 않다"고 판단되어 바디가 제거되지 않고 남는다.

### 해결 방향

바디 제거 스크립트에 **색상 허용 오차(tolerance)** 파라미터를 추가한다.

```txt
tolerance = 0  → 완전 일치만 제거 (가장 엄격)
tolerance = 5  → RGB 각 채널에서 ±5 이내면 제거 (권장 시작값)
tolerance = 10 → 넓은 허용 범위 (의상 외곽선도 지워질 위험)
```

### 주의사항

- tolerance가 너무 높으면 의상 외곽선이나 그림자도 바디로 오인되어 지워진다.
- tolerance가 너무 낮으면 미세하게 변형된 바디 픽셀이 잔류한다.
- 최적값은 실제 인페인팅 결과를 보면서 조절해야 한다.
- 이 파라미터는 `remove_body.py`에 반드시 포함되어야 한다.
- STEP 11 테스트 시 tolerance 값별 결과를 기록한다.

### 추가 방어선

tolerance만으로 부족할 경우 아래 보조 방법을 검토한다.

- 스킨톤 색상 범위를 직접 지정하여 피부색 픽셀을 추가 제거
- preserve_mask의 반전을 이용하여 의상 영역 밖의 잔여 픽셀 일괄 제거
- 알파 채널 기반 정리 (투명 영역 주변의 고립된 불투명 픽셀 제거)

### 16.1 카테고리 정의

* **top (상의)**
  - 상의 단독 의상 데이터입니다.
  - **예시**: 티셔츠, 블라우스, 후드, 니트, 세일러 상의, 상의 몸통 파츠 등.
* **bottom (하의)**
  - 하의 단독 의상 데이터입니다.
  - **예시**: 바지, 반바지, 치마, 스커트, 하의 파츠 등.
* **outfit (한벌옷)**
  - 한벌옷 또는 상하의가 합쳐진 완성 코디 데이터입니다.
  - **예시**: 원피스, 세일러 원피스, 멜빵 세트, 잠옷 세트, 상하의 완성 코디, 드레스형 의상 등.

---

### 16.2 body template 규칙

바디 템플릿(body template)은 의상 생성 대상이나 학습용 의상 데이터가 아닙니다. 

* **허용 용도 (정형/보조용)**
  - 기준 바디 포즈 유지 및 피팅 위치 확인
  - `preserve_mask` 생성을 위한 피부 영역 분석
  - 정렬 검수 및 `body subtraction` QA 비교 배경
* **금지 용도 (AI 입력/학습용)**
  - 의상 레퍼런스로 활용
  - IP-Adapter reference 입력으로 사용
  - LoRA 의상 스타일 학습 데이터 포함
  - KSampler Inpainting의 생성 대상 자체로 설정

---

### 16.3 데이터 처리 규칙

* **원본 보존**: 원본 의상 샘플 파일은 절대 덮어쓰거나 직접 수정하지 않습니다.
* **재귀적 스캔**: top / bottom / outfit 폴더를 재귀적으로 스캔하여 구조를 분석합니다.
* **속성 수집**: 개별 이미지의 크기, RGBA 채널 여부, 투명 배경 여부를 분석 및 기록합니다.
* **예외 처리**: 읽을 수 없거나 손상된 깨진 파일은 스캔 시 감지하여 별도 에러 목록으로 격리합니다.
* **메타데이터 중심**: 폴더를 지나치게 하위로 세분화하기보다 CSV/JSON 파일 태그를 통해 의상 타입과 이름을 체계적으로 관리합니다.
